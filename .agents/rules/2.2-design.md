# 2.2 — Design System & Architecture

## Purpose

Define UI architecture, component patterns, and design system rules that ensure **visual excellence**, accessibility, and maintainability across frontend and mobile projects. Every UI MUST look professional, modern, and enterprise-grade — not cookie-cutter or student-project quality.

## Design Tokens (ABSOLUTE)

Every project MUST define design tokens before building UI:

### Minimum Token Set

```css
/* colors */
--color-primary: ...;
--color-secondary: ...;
--color-background: ...;
--color-surface: ...;
--color-text: ...;
--color-text-muted: ...;
--color-border: ...;
--color-error: ...;
--color-success: ...;
--color-warning: ...;

/* typography */
--font-family-sans: ...;
--font-family-mono: ...;
--font-size-xs: ...;
--font-size-sm: ...;
--font-size-base: ...;
--font-size-lg: ...;
--font-size-xl: ...;
--font-size-2xl: ...;

/* spacing */
--space-1: 0.25rem;
--space-2: 0.5rem;
--space-3: 0.75rem;
--space-4: 1rem;
--space-6: 1.5rem;
--space-8: 2rem;

/* radii */
--radius-sm: ...;
--radius-md: ...;
--radius-lg: ...;
--radius-full: 9999px;

/* shadows */
--shadow-sm: ...;
--shadow-md: ...;
--shadow-lg: ...;
```

- Hardcoded color values in components FORBIDDEN — use tokens
- Inline magic spacing values FORBIDDEN — use spacing scale

> **Tailwind CSS projects:** Define design tokens in `tailwind.config.ts` → `theme.extend` and access via Tailwind classes. The CSS custom properties above serve as the LOGICAL reference — map them to Tailwind's theme configuration. Do NOT create both raw CSS tokens AND Tailwind config tokens.

## Component Rules (ABSOLUTE)

- One component per file
- Components MUST be pure where possible (deterministic render from props)
- Business logic MUST NOT live in components — extract to hooks or services
- Components MUST NOT directly call APIs — use React Query hooks
- Props MUST be typed with TypeScript interface (no inline types)
- Default exports FORBIDDEN — use named exports

### Component Composition Pattern

```typescript
// Prefer composition over configuration
// ✅ Good
<Card>
  <Card.Header>Title</Card.Header>
  <Card.Body>Content</Card.Body>
</Card>

// ❌ Bad
<Card title="Title" body="Content" headerVariant="large" />
```

## Accessibility (ABSOLUTE)

- Minimum touch target: 44x44px
- Keyboard navigation REQUIRED for all interactive elements
- Screen reader compatibility REQUIRED
- Color contrast MUST meet WCAG AA (4.5:1 for text, 3:1 for large text)
- Focus indicators MUST be visible
- Placeholders MUST NOT replace labels
- Hover-only interactions FORBIDDEN
- `aria-label` REQUIRED for icon-only buttons
- Form inputs MUST have associated labels

## Responsive Design (ABSOLUTE)

- Mobile-first REQUIRED
- Breakpoint system REQUIRED:

```css
--breakpoint-sm: 640px;
--breakpoint-md: 768px;
--breakpoint-lg: 1024px;
--breakpoint-xl: 1280px;
```

- Horizontal scroll on mobile FORBIDDEN (unless explicitly justified)
- Content MUST be readable without horizontal scrolling at 320px width
- Images MUST be responsive (max-width: 100%)
- Font sizes MUST be fluid or responsive

## Animation & Motion (AUTHORITATIVE)

- Animations ONLY if UX-justified (feedback, state change, attention)
- **State feedback transitions (hover, focus, active) are always UX-justified** — they provide essential user feedback and MUST be present on all interactive elements
- Duration: 150-300ms for micro-interactions, 300-500ms for transitions
- `prefers-reduced-motion` MUST be respected
- Decorative animations MUST be disabled when reduced motion is preferred
- Layout-shifting animations FORBIDDEN

## Iconography (AUTHORITATIVE)

- One icon library per project (Lucide, Heroicons, or Phosphor recommended)
- Mixing icon libraries FORBIDDEN
- Icons MUST have accessible labels when used without text
- Icon sizes MUST follow the spacing scale

## Dark Mode (AUTHORITATIVE)

If dark mode is supported:

- Design tokens MUST support both light and dark themes
- Use CSS custom properties for theme switching
- Images MUST be tested in both themes
- Respect `prefers-color-scheme` system preference
- Manual toggle MUST persist user choice

## Visual Quality Standards (ABSOLUTE)

Every frontend and mobile project MUST meet these visual quality standards. These apply regardless of project tier — visual excellence is non-negotiable.

### Color Palette

- Curated, cohesive color palette REQUIRED — the palette MUST have personality and intent
- Minimum palette: primary + secondary + 2 accent colors + semantic colors (error, success, warning, info)
- Raw basic CSS colors FORBIDDEN (`red`, `blue`, `green`, `gray`) — use curated, specific values
- HSL-based color definitions RECOMMENDED for consistency and easy theme generation
- Color palette MUST be defined during planning (see `2.1-planning.md` UI/UX Direction)
- Neutral scale MUST have at least 5 shades (50, 100, 200, ... 900) for depth and flexibility

### Typography

- Professional font REQUIRED — default browser fonts FORBIDDEN
- Recommended font families: Inter, Plus Jakarta Sans, Outfit, DM Sans, Geist, Satoshi
- Mobile recommended: System font stack is acceptable for performance, but MUST feel polished
- Weight hierarchy REQUIRED: regular (400) → medium (500) → semibold (600) → bold (700)
- Line-height ratios: 1.4–1.6 for body text, 1.1–1.3 for headings
- Letter-spacing: slight negative for large headings (-0.01em to -0.02em), default for body
- Maximum 2 font families per project (1 sans + 1 mono recommended)
- Font sizes MUST follow the design token scale — no arbitrary sizes

### Visual Hierarchy

- Every page/screen MUST have a clear focal point — the user's eye should know where to start
- Establish hierarchy through: size contrast, weight contrast, color contrast, and spatial positioning
- Headings MUST be visually distinct from body text (size + weight difference)
- Primary actions MUST be visually prominent — secondary actions MUST be visually subordinate
- Information architecture MUST be scannable — use grouping, spacing, and labeling

## Component Visual Quality (ABSOLUTE)

### Cards

- Cards MUST have: proper border/shadow, border-radius, internal padding, content spacing
- Hover state REQUIRED on clickable cards (subtle shadow lift, border color, or background shift)
- Card content MUST have consistent spacing between title, body, and footer
- Card grids MUST have consistent gap spacing

### Forms

- Form fields MUST be visually grouped by context (personal info, payment, etc.)
- Labels REQUIRED and MUST be visually associated with inputs (above, not just placeholder)
- Inline validation REQUIRED — show error state on the specific field
- Disabled state MUST be visually distinct (reduced opacity + cursor change)
- Input focus state MUST be visible and styled (not just browser default blue)
- Form actions (submit, cancel) MUST have clear visual hierarchy (primary button vs secondary)

### Tables & Lists

- Row hover state REQUIRED for interactive tables
- Visual row separation REQUIRED (zebra striping, borders, or dividers)
- Sort indicators REQUIRED on sortable columns
- Empty state REQUIRED — no blank tables. Show message + illustration or icon + action suggestion
- Pagination or infinite scroll REQUIRED for data > 20 rows

### Empty States

- Blank screens FORBIDDEN — every empty state MUST be designed
- Empty state MUST include: icon or illustration + descriptive text + suggested action (CTA button)
- Tone MUST be friendly and helpful, not just "No data found"

### Loading States

- Skeleton screens PREFERRED over generic spinners
- Skeleton MUST match the approximate shape and layout of expected content
- Full-page spinners FORBIDDEN for partial data loads — use inline or section-level skeletons
- Loading state MUST appear within 200ms if content hasn't loaded

### Error States

- Raw error codes or stack traces FORBIDDEN in user-facing UI
- Error messages MUST be: human-readable, actionable, and include recovery steps
- Error boundaries MUST show a styled fallback — not a blank screen or browser error
- Network error states MUST offer retry capability

## Page Layout Quality (ABSOLUTE)

### Whitespace

- Generous whitespace REQUIRED — cramped layouts FORBIDDEN
- Use the spacing token scale consistently — do not mix arbitrary pixel values
- Section padding MUST be at least `--space-8` (2rem) or equivalent
- Content MUST NOT touch the edges of its container — internal padding REQUIRED

### Grid System

- Consistent column-based grid REQUIRED for page layout
- Content area MUST have a max-width for readability (recommended: 1200-1400px for desktop)
- Prose content (paragraphs, articles) MUST have a max-width of ~65-75 characters (600-700px)
- Grid gaps MUST use the spacing token scale

### Section Rhythm

- Long pages MUST have clear visual section breaks (spacing, background changes, dividers)
- Alternating visual patterns RECOMMENDED for long scrolling pages (e.g., alternate background)
- Related content MUST be visually grouped — unrelated content MUST be visually separated

### Navigation

- Sticky/fixed navigation RECOMMENDED for pages with significant scroll depth
- Active state indicator REQUIRED on current navigation item
- Navigation MUST be accessible at all times — user MUST NOT feel lost
- Mobile navigation MUST collapse to a pattern appropriate for the platform (hamburger, bottom tabs)

## Polish & Refinement (ABSOLUTE)

### Micro-interactions

- Hover state REQUIRED on ALL interactive elements (buttons, links, cards, list items)
- Focus state REQUIRED and MUST be visible (keyboard accessibility)
- Active/pressed state REQUIRED on buttons (subtle scale or color shift)
- Transitions MUST be smooth (use `transition` with 150-200ms `ease-out`)
- State changes MUST feel responsive — no abrupt visual jumps

### Shadow & Depth

- Layered elevation system REQUIRED:
  - Level 0: flat (page surface)
  - Level 1: subtle shadow (cards, raised elements)
  - Level 2: medium shadow (dropdowns, popovers)
  - Level 3: strong shadow (modals, dialogs)
- Shadow values MUST use the design token scale (`--shadow-sm`, `--shadow-md`, `--shadow-lg`)
- Consistent shadow direction across all elements (recommended: light source from top-left)

### Borders & Dividers

- Borders MUST be subtle (1px, using `--color-border` token)
- Hard black borders (`1px solid black`) FORBIDDEN
- Dividers MUST use the border color token — not arbitrary gray values
- Border-radius MUST be consistent — use the radius token scale

### Gradients

- Subtle, directional gradients ALLOWED for backgrounds and surfaces
- Rainbow or multi-color gradients FORBIDDEN unless brand-appropriate
- Gradient direction SHOULD be consistent across the UI

### Icon Consistency

- All icons MUST use the same weight/stroke across the entire project
- Icon sizes MUST be optically consistent (aligned to 4px grid: 16px, 20px, 24px)
- Icons MUST be vertically centered with adjacent text

## Anti-Patterns FORBIDDEN (ABSOLUTE)

The following patterns indicate low-quality UI and are FORBIDDEN:

- Generic gray-on-white layouts without brand identity or color personality
- Unstyled HTML elements (raw `<button>`, `<input>`, `<select>` without styling)
- Default browser fonts (Times New Roman, system serif)
- Inconsistent spacing between similar elements
- Missing hover/focus/active states on interactive elements
- Hard pixel borders (`1px solid black` or similar)
- Placeholder-only forms without proper labels
- Walls of text without visual breaks, headings, or whitespace
- Raw error messages or stack traces in user-facing UI
- Blank/empty screens without designed empty states
- Full-page spinners for partial content loads
- Layout shifts during loading
